# Spec Document — business ↔ code field-level documentation

A native port of Auto Spec Kit's `/document`. Produce accurate, citable documentation that a
mixed business+technical audience can trust. Use ONLY the provided context (KB + real
source); if something isn't there, write `(not found in codebase)` — never invent.

## Input
The topic to document (a feature, entity/model, or module).

## Gather context
1. Load relevant `knowledge-base/` docs plus, by default, the business/domain/architecture
   docs and any matching `knowledge-base/modules/<module>.md`.
2. Discover and read the **relevant source files** (Grep/Glob) for field-level grounding —
   you must name real classes, functions, and fields.

## Output structure (Markdown; use TABLES for every mapping)
```
# <Clear document title>
_Scope: <topic> · generated by Spec Kit_

## 1. Business Overview
What it does, who uses it, the business value/problem it solves.

## 2. Business ↔ Code Field Mapping
For EACH relevant entity/model/DTO, a table over every important field:
| Field | Type | Business Meaning | Validation / Business Rule | Source (file · class.field / DB column) |
Be exhaustive; cite real names.

## 3. Business Flows (mapped to code)
Step-by-step; for each step name the file + function that implements it.

## 4. Business Rules & Invariants
| Rule | Where enforced (file · function) | Severity |

## 5. API / Entry Points (if any)
| Method | Path | Purpose | Auth |

## 6. Assumptions & Gaps
Anything inferred, ambiguous, or "(not found in codebase)".
```

## Save
Write to `spec-kit-sessions/documents/<topic-slug>-<date>.md`. (The original also emitted a
styled HTML with rendered Mermaid; offer to generate an HTML version if the user wants one.)
If `knowledge-base/` is missing, document from source directly but recommend `/namht-scan`
for richer, business-grounded output.
